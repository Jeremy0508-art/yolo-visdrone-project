# 《计算机工程与应用》投稿就绪执行计划

本文档是后续工作的执行主线。这里的“100%”指把项目推进到“材料完整、实验公平、证据可追溯、论文结构符合中文期刊审稿预期、可以正式投稿”的状态；它不等同于承诺期刊一定录用。后续实验、论文重写、图表更新、README 维护和 GitHub 同步均按本文档逐项推进。

## 1. 投稿目标

- 目标期刊：《计算机工程与应用》。
- 论文方向：面向 VisDrone2019-DET 无人机航拍场景的小目标检测方法改进与实验分析。
- 项目主线：以 YOLO11n 为基础，围绕 P2 高分辨率检测分支、CoordAttention、960 输入分辨率和小目标增强策略，系统分析精度、速度、复杂度和小目标检测能力之间的关系。
- 投稿材料目标：形成期刊长文稿件、完整实验矩阵、图表源文件、复现实验命令、证据审计、GitHub 项目展示页和导师汇报材料。

## 2. 期刊写作门槛

参考《计算机工程与应用》及同类中文核心/科技期刊中 YOLO 目标检测论文的常见写法，当前稿件需要达到以下门槛：

1. 问题定义清楚：说明无人机航拍图像中目标小、密集、遮挡、尺度变化大和类别相似带来的检测困难。
2. 方法动机闭环：每个模块必须对应一个具体问题，不能只写“加入某模块提高精度”。
3. 公平对比充分：同输入分辨率、同训练轮次、同数据集划分下比较 baseline、改进模型和不同 YOLO 版本。
4. 消融实验可信：P2、CoordAttention、960 输入和 SmallObjAug 要能分别解释贡献或负向结果。
5. 小目标证据充足：除总体 mAP 外，还要有目标尺度分布、尺度分组召回、可视化案例和失败案例。
6. 工程指标完整：参数量、GFLOPs、模型大小、单图推理时间、FPS 和精度-速度折中图要统一呈现。
7. 论文体量充足：引言、相关工作、方法、实验、讨论和结论应按期刊长文展开，而不是短报告式写法。
8. 证据可追溯：所有数值必须来自 `runs/`、`runs/logs/`、`paper/tables/` 或官方评测返回结果。

网络检索到的同类 YOLO 小目标论文通常会报告模块结构图、消融实验、多数据集或多模型对照、复杂度/FPS、检测可视化与失败场景。例如《计算机工程》2025 年一篇 YOLOv8 小目标检测论文在摘要中同时报告了模块改进、VisDrone2019 提升、多数据集泛化和图表化实验分析；《计算机工程与应用》相关条目和引用记录中也能看到无人机小目标、YOLOv8、轻量化检测等主题。本文后续写作应对齐这种论证密度，但所有本项目数值只使用自己复现或已有真实文件。

## 5. 结果判读原则

1. 不把 YOLOv8n-640 与 YOLO11n-P2-CA-960 当成单因素公平对比，因为架构和输入分辨率都不同。
2. 如果 YOLO11n-960 接近主方法，则论文重点转为“高分辨率输入是主要贡献，P2/CA 提供额外结构增益”。
3. 如果 YOLO11n-P2-960 明显优于 YOLO11n-960，则 P2 分支可作为结构贡献重点。
4. 如果 YOLO11s-960 超过主方法，则论文必须强调轻量化参数效率和速度-精度折中，不能写“全面最优”。
5. 如果 SmallObjAug 结果不佳，则作为负向消融保留，说明简单增强并不一定优于结构与输入分辨率设计。
6. 如果所有新增对照都削弱当前主张，则论文转向“系统评估与折中分析”，避免强行宣称方法优势。

## 6. 论文重写路线

### 6.1 摘要

摘要必须包含：

- 任务背景与痛点。
- 方法核心设计。
- 数据集与实验协议。
- 主要真实结果。
- 速度或复杂度折中。

### 6.2 引言

引言扩展为 5 个逻辑段：

1. 无人机目标检测应用背景。
2. VisDrone 小目标、密集和遮挡挑战。
3. YOLO 系列在实时检测中的优势与不足。
4. 当前方法从高分辨率特征、坐标注意力和输入分辨率三个角度改进。
5. 本文贡献列表。

### 6.3 相关工作

至少覆盖：

- YOLO 系列目标检测。
- 无人机航拍目标检测。
- 小目标检测与多尺度特征融合。
- 注意力机制在检测中的应用。
- 轻量化检测与速度-精度折中。

### 6.4 方法

方法部分按“问题-设计-作用”重写：

- P2 分支：解决小目标在深层特征图中细节丢失。
- CoordAttention：增强坐标方向位置信息建模。
- 960 输入：缓解输入缩放导致的小目标不可见。
- SmallObjAug：作为数据层面增强尝试，并按真实结果讨论。

### 6.5 实验

实验部分必须包含：

- 数据集与划分。
- 训练配置。
- 评价指标。
- 主结果表。
- 公平分辨率对照表。
- 消融实验表。
- 复杂度与速度表。
- 类别级分析。
- 尺度分组分析。
- 可视化与失败案例。

### 6.6 讨论与局限

讨论部分必须解释：

- 为什么 960 输入有效但带来速度成本。
- P2 与 CA 是否值得保留取决于公平对照结果。
- 哪些类别仍容易混淆。
- 方法在部署端可能受显存、速度和输入分辨率限制。

## 8. 证据与审计流程

每次服务器实验完成后按以下流程处理：

1. 运行 `tools/check_cea_server_status.ps1` 检查是否达到 100 epoch。
2. 运行 `tools/sync_cea_server_results.ps1`，只同步完整实验。
3. 运行 `tools/export_paper_tables.py` 更新结果表。
4. 对新增模型补测速度和复杂度。
5. 运行 `python tools/check_paper_consistency.py`。
6. 运行 `python tools/audit_submission_readiness.py`。
7. 重新编译 PDF。
8. 更新 `paper/CEA_EXECUTION_LOG.md`。
9. 提交并推送 GitHub。
