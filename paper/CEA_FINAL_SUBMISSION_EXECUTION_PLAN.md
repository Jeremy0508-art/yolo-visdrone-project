# 《计算机工程与应用》投稿级最终执行计划

本文档是后续推进本项目的最高优先级路线图。这里的“100%实现投稿中刊”指把项目推进到可以正式提交《计算机工程与应用》审稿系统的材料状态：实验矩阵完整、数值证据可追溯、论文论证达到中文期刊长文要求、排版和复现材料经审计通过。它不承诺期刊一定录用，但要求我们把可控部分做到投稿级。

## 1. 最终目标

- 目标期刊：《计算机工程与应用》。
- 研究对象：VisDrone2019-DET 无人机航拍目标检测。
- 技术主线：以 Ultralytics YOLO11n 为基线，围绕 P2 高分辨率检测分支、CoordAttention、960 输入分辨率和小目标友好增强策略，构建并验证面向航拍小目标的轻量化检测方法。
- 投稿状态目标：形成可提交 PDF/LaTeX 稿件、完整图表、真实实验结果表、复现实验命令、证据审计文件、GitHub 项目说明和导师汇报材料。
- 约束原则：不编造数值，不使用未完成训练结果作为论文结论，不把不同训练协议下的模型做不公平强结论比较。

## 2. 期刊化质量门槛

1. 论文必须从“项目报告”升级为“期刊论文”：问题定义、方法动机、实验协议、结果分析、失败案例和局限性都要展开。
2. 方法创新不能只写成模块堆叠，必须建立“航拍小目标问题 - 结构设计 - 实验证据”的闭环。
3. 对比实验必须能回答审稿人最可能追问的问题：
   - 性能提升是否只是来自 960 输入分辨率？
   - P2 分支在同分辨率下是否仍有贡献？
   - CoordAttention 是否带来稳定增益？
   - 与 YOLOv5n、YOLOv8n、YOLO11n、YOLO11s 等常见轻量模型相比，本文方法的优势或折中在哪里？
4. 小目标主题必须有专项证据：目标尺度分布、尺度分组指标、类别混淆、可视化示例和失败案例。
5. 工程指标必须完整：参数量、GFLOPs、模型大小、推理延迟、FPS、精度-速度折中。
6. 参考文献需要覆盖近年 YOLO、无人机目标检测、小目标检测、多尺度特征融合、注意力机制和轻量化检测相关工作。
7. 最终 PDF 不能出现占位标记、乱码、异常断词、未完成实验名称或无证据结论。

## 5. 服务器无人值守执行规则

1. 每次检查服务器只更新 `paper/cea_server_status_snapshot.md`、`paper/tables/cea_server_status_history.csv` 和 `paper/cea_server_progress_report.md`。
2. 未达到 100 epoch 的训练只算进度，不进入 `paper/tables/main_results.csv`、论文正文或结论。
3. 某个服务器实验完成后，按以下顺序处理：
   - 运行 `tools/check_cea_server_status.ps1` 确认完整性。
   - 运行 `tools/sync_cea_server_results.ps1` 同步完整结果。
   - 重新生成 `paper/tables/*.csv`。
   - 为新增模型补测复杂度、速度、类别级指标和必要的尺度分组指标。
   - 运行 `python tools/run_paper_audits.py`。
   - 只在审计通过后更新论文正文。
4. 如果服务器中断，优先从已有 checkpoint 或日志恢复；无法恢复时，只保留已经完整完成的实验。

## 6. 论文重写路线

### Phase A：结果就绪前的本地强化

- 清理并完善计划文档、执行日志、README、paper README 和导师汇报材料。
- 检查审计脚本覆盖范围，确保所有论文材料能自动检查。
- 扩充参考文献候选清单，不把未核验文献写成正式引用。
- 先不改动主结论，等待公平对照实验完成。

### Phase B：服务器结果同步与结果判读

- 同步所有完整 100 epoch 实验。
- 更新主结果表、公平分辨率表、主流 YOLO 对照表、消融表、复杂度表和速度表。
- 建立结果判读规则：
  - 若 YOLO11n-960 接近主方法，论文强调高分辨率输入是主要贡献，P2/CA 是额外结构增益。
  - 若 YOLO11n-P2-960 明显优于 YOLO11n-960，论文强化 P2 对小目标的贡献。
  - 若 YOLO11s-960 优于主方法，论文改写为轻量化折中优势，而不是绝对最优。
  - 若所有新增对照削弱主张，论文转为系统评估和实证分析型稿件，保留真实负结果。

### Phase C：期刊长文重写

- 摘要：写清背景、方法、数据集、真实结果和速度/复杂度折中。
- 引言：扩展为无人机小目标检测问题、YOLO 系列不足、本文设计动机、贡献总结。
- 相关工作：覆盖 YOLO、无人机检测、小目标检测、多尺度融合、注意力机制、轻量化检测。
- 方法：按 P2、CoordAttention、960 输入、小目标增强分别解释问题来源和设计作用。
- 实验：分为实验设置、主结果、公平分辨率对照、主流 YOLO 对照、消融、复杂度速度、类别分析、尺度分析、可视化、失败案例。
- 讨论：解释优势边界、速度成本、负向结果、实际部署限制。
- 结论：只总结已证实结论，避免承诺式表述。

### Phase D：图表与格式定稿

- 所有表格统一小数位、指标名称、数据来源列和模型命名。
- 生成或更新方法结构图、尺度分布图、精度速度折中图、可视化对比图。
- 统一 LaTeX 浮动体位置，避免图表进入结论或参考文献。
- 清理参考文献乱码、异常断词和未核验条目。
- 编译 PDF 并做人工页检。

### Phase E：投稿前审计

- 运行全文搜索：占位标记、乱码字符、未完成实验名、旧结果数值。
- 运行所有本地审计：`python tools/run_paper_audits.py`。
- 检查每个正文数值是否能追溯到 CSV、log、args.yaml、results.csv 或脚本输出。
- 检查 GitHub README 是否是项目介绍，而不是进度汇报。
- 准备导师汇报材料和投稿材料清单。

## 8. 每日无人值守优先级

1. 检查服务器队列是否有完整 100 epoch 新结果。
2. 若有完整结果，执行同步、导表、补测、审计、论文更新和 GitHub 提交。
3. 若无完整结果，继续强化本地论文材料、参考文献、审计脚本和 README。
4. 每次提交都必须写清楚改动范围，不遗漏 README、论文、脚本、表格和审计材料的变化。
5. 不为“看起来更好”而改写或隐藏真实负结果。

## 9. 最终验收标准

项目达到投稿状态时，应同时满足：

- 公平对照实验全部完成或有明确、合理、可解释的缺失说明。
- 正文所有结果来自真实文件，且能通过审计追溯。
- 论文篇幅、结构和论证密度接近中文期刊长文，而非课程报告或短论文。
- 图表完整、编号连续、引用正确、无遮挡和越界。
- 参考文献格式统一，无乱码和异常断词。
- PDF、LaTeX、CSV、图表、命令、README、审计文件全部同步到 GitHub。
- 导师能够通过 README、PDF 和汇报材料快速理解研究问题、方法、实验矩阵、主要结果和风险边界。
