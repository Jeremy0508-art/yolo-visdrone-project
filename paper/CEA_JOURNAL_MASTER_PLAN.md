# 《计算机工程与应用》投稿主控计划

本文档是后续工作的主线计划。目标不是承诺论文一定录用，而是把项目推进到“可以正式投稿《计算机工程与应用》且经得起审稿追问”的状态。后续新增实验、论文重写、图表生成、README 更新和 GitHub 同步都按本文档执行。

## 1. 投稿目标

- 目标期刊：《计算机工程与应用》。
- 论文类型：面向无人机航拍小目标检测的 YOLO 改进与系统实验论文。
- 投稿状态目标：完成真实实验矩阵、证据审计、期刊化长文稿、图表源文件、复现命令和 GitHub 项目展示。
- 质量目标：从“课程/会议短文”升级为“中文期刊可审稿稿件”，正文论证、实验对比和可复现材料均达到期刊论文标准。

期刊公开投稿说明提到一般论文建议 7500 字以上，论文应包含题名、作者、单位、摘要、关键词、基金资助、正文和参考文献等基本部分。后续稿件按长文组织，不再按短论文压缩。

参考页面：

- 《计算机工程与应用》投稿须知：`https://jsjgcyyy.juqk.net/buy/`
- 《计算机工程与应用》编辑部页面：`https://jsjgcyyy.juqk.net/`

## 2. 总体研究主张

论文主线建议固定为：

> 面向 VisDrone 无人机航拍场景中目标尺度小、分布密集、遮挡严重和类别易混淆的问题，构建一种融合高分辨率 P2 检测分支、坐标注意力与高分辨率输入策略的轻量化 YOLO11n 检测方法，并通过公平分辨率对照、主流 YOLO 基线、消融实验、尺度分组分析、速度复杂度统计和可视化案例系统验证其有效性。

建议方法名称：

- 英文简称：`HRPCA-YOLO11n`
- 中文名称：融合高分辨率 P2 分支与坐标注意力的轻量化 YOLO11n 无人机小目标检测方法

注意：最终名称需要根据全部对照实验结果调整。如果后续公平对照显示结构优势不明显，论文主张应转向“系统评估和精度-速度折中分析”，不能硬写“全面优于”。

## 4. 关键差距

| 差距 | 审稿风险 | 解决方式 |
| --- | --- | --- |
| 创新点容易被认为是模块堆叠 | 审稿人可能认为 P2、CA、960 输入都是常规组合 | 用“问题-设计-证据”重写方法动机，并用尺度分组结果支撑小目标主题 |
| 960 输入贡献未被公平隔离 | 无法说明性能提升来自结构还是分辨率 | 补齐 YOLO11n-960、YOLO11n-P2-960、YOLOv8n-960、YOLO11s-960 |
| 主流方法对比不足 | 很难证明相对常见 YOLO 模型有竞争力 | 补齐 YOLOv5n、YOLOv8n、YOLO11n、YOLO11s 等同训练协议对比 |
| 小目标专项指标不足 | “小目标检测”主题支撑不够 | 增加尺度分布、scale-group recall/precision、失败案例分析 |
| 论文体量偏短 | 不符合期刊长文论证深度 | 扩写引言、相关工作、方法细节、讨论和局限性 |
| 格式尚非期刊终稿 | 投稿前可能返工 | 准备 LaTeX 长文稿和 Word 迁移材料，最终按期刊模板排版 |
| 参考文献仍需加强 | 文献覆盖不足会削弱选题依据 | 增加近三年 YOLO、小目标检测、无人机检测、注意力机制相关论文 |

## 6. 证据规则

1. 论文中所有指标必须来自真实文件：`runs/`、`runs/logs/`、`paper/tables/`、官方评测返回结果或明确记录的脚本输出。
2. 不完整训练结果不得进入论文主表。
3. 服务器实验必须达到完整 100 epoch 后才同步为论文证据。
4. 外部论文中的数值只能作为“文献报道结果”，不能与本项目复现实验混为同一协议对比。
5. VisDrone test-dev 官方 AP 只有在官方平台返回后才能写入正文。
6. README 面向教授和读者介绍项目，不写过程性状态、上传清单或内部进度汇报。

### Phase 2：等待并同步公平对照实验

目标：补齐期刊审稿最可能追问的公平对照。

### Phase 3：结果判读与论文主张修正

目标：根据真实结果决定论文怎么写，而不是提前锁死结论。

任务：

产出：

- 公平对照分析表
- 精度-速度-复杂度综合表

## 8. 风险预案

| 可能结果 | 应对策略 |
| --- | --- |
| YOLO11n-960 已接近本文方法 | 论文改写为“高分辨率输入是主要贡献，P2/CA 提供额外小幅增益”，避免夸大结构创新 |
| YOLO11s-960 超过本文方法 | 强调本文方法的轻量化参数效率，或把 YOLO11s-960 作为性能上限讨论 |
| YOLOv8n-960 接近本文方法 | 以同协议复现实验证明本文方法在 YOLO11n 体系下的折中优势，不做跨版本绝对优势夸大 |
| SmallObjAug 表现不佳 | 保留为负向消融，说明简单增强策略不一定优于结构与分辨率设计 |
| 服务器实验中断 | 从已有完整 checkpoint 或日志恢复；无法恢复时只使用已完成证据 |
| 官方 test-dev 不可用 | 只报告验证集结果，不写官方 AP |
